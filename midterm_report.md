## 一、项目介绍

### 1.1 研究背景

近年来，以 GPT 系列、通义千问、文心一言为代表的大语言模型（Large Language Model，LLM）在自然语言理解、逻辑推理、任务规划等方面取得跨越式进展，为人工智能从"感知智能"迈向"决策智能"奠定了基础。在此背景之上，智能体（Agent）技术成为大模型落地应用的关键方向：通过为大模型赋予工具调用、自主规划、多轮迭代能力，智能体能够突破大模型自身的知识边界，对接外部系统与专业工具，从而完成复杂的现实世界任务。与此同时，AI for Science（人工智能赋能科学研究）被广泛视为加速科学发现、提升科研效率的重要技术手段，科学研究场景下普遍存在大量重复性、流程化的工作，包括数值计算、统计分析、实验数据可视化等，迫切需要智能化、自动化的辅助工具支持。

然而，当前主流的智能体框架与工具调用技术在通用场景中已被广泛验证，却在科学研究这一垂直场景下存在明显的适配性缺陷：其一，缺乏面向科学领域的统一工具抽象规范，不同科研库接口差异极大，集成成本高；其二，对专业科研任务的理解与拆解能力不足，难以将模糊的科研问题拆解为严谨、可执行的工具调用序列；其三，工具调用的容错性与结果可靠性不足，存在参数幻觉、调用逻辑错误、数值精度随意截断等问题，无法满足科研工作的严谨性要求。

### 1.2 研究目标

本研究旨在系统性地研究面向科学场景的大语言模型智能体工具调用优化问题。结合指导教师与盲审专家"应强化核心算法创新、补充关键技术模块算法细节"的指导意见，本课题在开题阶段的工程集成方案基础上，进一步将研究目标聚焦为：提出两个可独立消融、形式化可分析的算法模块，作为本论文的核心贡献点：

1. **RATS（Retrieval-Augmented Tool Selection，检索增强工具选择算法）**：针对工具库规模增长后 context 膨胀、LLM 选错率上升的问题，设计基于 embedding 语义检索加规则重排的轻量工具选择算法；
2. **DAGPlanner（结构化 DAG 任务拆解算法）**：针对长依赖链科学任务拆解薄弱的问题，设计一种基于有向无环图（DAG）schema 的结构化规划器，通过纯代码校验与错误驱动的重规划实现收敛。

在此之上，完成一套支撑性的科学计算工具库、科学场景专用 Prompt 工程规则集、以及用于对比验证的评估框架与基准测试集。

### 1.3 研究意义

本研究在算法层面填补了"面向科学场景的轻量工具检索"与"结构化任务拆解"两项空白，弥补了通用智能体框架在科学场景下的适配性缺陷，为科研领域智能体系统的设计提供可参考的实现范式；在应用层面，系统的模块化与可扩展设计使其能够快速迁移至生物信息、材料科学、环境科学等细分科研领域，具备较强的落地潜力。

## 二、本人任务

作为本课题唯一学生，毕业设计的全部工程实现、算法设计、实验与论文撰写工作均由本人独立承担。具体任务拆解如下：

- **T1 标准化科学工具库设计与实现**：基于 `BaseTool` 抽象基类统一工具接口规范，涵盖参数声明、执行逻辑、返回结构（`ToolResult`）、异常处理等要素。基于 NumPy / SciPy / Matplotlib 封装 12 个高频科研工具，覆盖数值计算（矩阵运算、数值积分、曲线拟合、方程求解）、统计分析（描述性统计、假设检验、线性回归、相关性分析）、数据可视化（折线图、散点图、条形图、热力图）三大类别；为每个工具编写单元测试，并通过 `to_openai_schema()` 方法对接 LLM Function Calling 协议。
- **T2 核心算法模块的设计、实现与形式化分析**：完成 RATS 与 DAGPlanner 两个算法模块的从零设计到落地。RATS 侧重"选什么"，包含离线预计算、在线向量检索与规则重排三个阶段；DAGPlanner 侧重"怎么组织"，提出可纯代码校验的 Plan Schema，实现拓扑序执行与错误驱动重规划。两个模块给出形式化伪代码、复杂度分析与收敛性讨论，作为论文核心章节的素材。
- **T3 Prompt 工程优化体系**：针对科学场景提炼出 O1 结果合理性检查、O2 数值计算规范、O3 开场规划模板、O4 Few-shot 示例注入共 4 条规则，作为算法模块运行时的必要辅助约束。
- **T4 评估框架与基准构建**：设计三维评估判分机制（数值正确性、文件正确性、工具覆盖性），实现用例 JSON 定义、执行器、自动化评测报告生成全链路。完成 54 题科学任务测试集（5 大类别 × 3 档难度），以及 9 配置对比实验框架，其中包含 CoT、Plan-and-Solve、Reflexion 三个业界常用基线。
- **T5 论文撰写与答辩准备**：完成毕业论文的撰写、多轮修订、答辩 PPT 制作与最终定稿。

## 三、工作计划

### 3.1 总体节奏与阶段划分

本课题在总体节奏上仍遵循开题阶段确立的"由调研与基建起步、以工程实现与评测闭环为中枢、以论文成稿与答辩为收口"的纵向推进方式。工程侧按自然顺序先完成可复现的数据与量尺，再开展方法迭代与对比实验，避免"先写结论后补数据"的论证风险。论文侧将优先保证两大算法模块的表述与实验证据一一对应，使方法章节、实现章节与实验章节能形成闭合链条。

### 3.2 工程与实验安排

在工程与实验层面，后续工作重点放在把已有实现固化为可对外讲述的研究贡献。一方面需要把 RATS 与 DAGPlanner 的输入输出、失败模式、降级路径与评估指标在代码与实验记录中保持口径一致，另一方面需要把多配置对比与消融结果整理成可写入论文的图表与文字说明。若发现个别用例在边界条件下仍出现轨迹不稳定，将以复现优先的方式补充诊断信息，而不是扩大无关改动面。

### 3.3 论文撰写与答辩准备

在论文撰写层面，将按"先搭骨架、后填血肉、再统一叙事"的方式推进。引言与相关工作将强调科学场景对精度、可解释性与可复现性的要求，并说明为何需要工具检索与结构化规划两类能力。方法章节将围绕 RATS 与 DAGPlanner 分别给出可独立阅读的形式化描述，并在必要时点明与 Prompt 规则之间的配合关系。系统实现章节将说明工具库、Agent 主循环、评测框架与配置矩阵如何组织，使读者能按模块理解工程结构。实验章节将以对比实验为主线，同时保留针对典型失败样例的定性分析，以支撑"鲁棒性、效率、可解释性"等多维度的讨论。在接近定稿时，将同步准备答辩材料，用架构图、关键表与案例说明把核心贡献讲清楚、讲可验证。

### 3.4 预期目标

预期在毕业设计结题时，系统侧保持工具库、算法模块、评测脚本的稳定可运行，论文侧形成结构完整、数据与代码相互印证的终稿。量化层面希望测试集与对比配置维持在一个足以支撑统计结论与消融讨论的规模，并在关键指标上体现本方案相对常用基线具有可解释的优势。质量层面更重视论证链条的完备性，包括实验设置是否足够透明、失败案例是否得到解释、局限性与未来工作是否表述诚实，从而满足学院对材料完整性与规范性的要求。

## 四、当前进度

截至 2026 年 4 月 23 日，课题的工程与实验部分已基本收尾，进入论文撰写阶段，完成情况如下。

### 4.1 基础工程（阶段 1-3，已完成）

环境层面，基于 Python 3.14.4 搭建开发环境，采用 venv 虚拟环境管理依赖；底层 LLM 选用通义千问 qwen-plus-latest 与 qwen-turbo-latest（OpenAI 兼容接口），涉及 embedding 时使用 text-embedding-v3；数值计算基于 NumPy 2.4、SciPy 1.17，绘图基于 Matplotlib 3.10。工具库层面，已完成 12 个工具的封装，统一继承 `BaseTool` 抽象基类，统一返回 `ToolResult(success, data, error, message)` 数据结构；为支持 LLM 传入字符串数学表达式的场景（如数值积分、方程求解），还实现了基于 AST 白名单的安全表达式求值模块。测试层面，已编写 45 个工具级单元测试（数值 18 + 统计 17 + 可视化 10），全部通过。Agent 层面，手写实现了 `ReActAgent` 主循环，采用 OpenAI Function Calling 协议加显式中文 Thought 的组合模式，并对 LLM 客户端做了超时控制与指数退避重试。

### 4.2 Prompt 工程优化（阶段 4，已完成）

针对科学场景下的典型失败模式，经过多轮实验与 case 级轨迹分析，提炼出 4 条 Prompt 规则：O1 结果合理性检查（拟合 $R^2$ 为负或超出 $[0,1]$、参数量级失配、返回值含 NaN/Inf 视为异常，必须切换方法）、O2 数值计算规范（3 个以上样本的统计量必须通过工具计算，禁止心算）、O3 开场规划模板（3 步以上任务第一轮必须写 `Plan:` 并逐步执行）、O4 Few-shot 示例注入（在 SYSTEM_PROMPT 末尾追加典型轨迹示例）。在原 46 题测试集上，Prompt 优化前后基线（v3 → v4）的对比实验显示：Ours 配置成功率从 93.5% 提升至 97.8%，同时工具调用次数相对 B1 基线减少约 8.8%，证明该组规则对科学场景具有针对性收益。

### 4.3 响应审核意见的算法深化（阶段 5-6，核心工作，已完成）

**RATS 算法模块**：实现于 `src/tool_selector/`，包含三个核心组件——`Embedder`（封装阿里云 OpenAIEmbedder，另提供 `HashEmbedder` 作为离线 fallback）、`EmbeddingCache`（按 `tool_name::desc_hash::model_id` 组织持久化缓存）、`RATS.retrieve()`（执行离线预计算、在线 top-K 检索与规则重排三阶段）。重排综合考虑 embedding 相似度（权重 0.6）、参数重叠度（0.2）、历史成功率（0.2）三项特征，并设置 `K_min=3` 作为保底机制。该模块通过 9 个单元测试（`tests/tool_selector/test_retriever.py`）全部绿灯。

**DAGPlanner 算法模块**：实现于 `src/agent/dag_planner.py` 与 `src/agent/dag_agent.py`。核心设计包括：一套可纯代码校验的 `Plan Schema`，每个步骤含 `id`、`goal`、`tool`、`args_template`（支持 `${sX.field}` 占位符）、`depends_on` 字段；一个 `PlanValidator` 覆盖 5 类失败模式——JSON Schema 不符、工具名不在候选集、依赖图有环、依赖变量未声明、占位符不在 depends_on 中；一个四阶段主循环 `Plan → Validate → Execute → Answer`，其中 Validate 阶段完全不调用 LLM，仅在失败时带诊断信息触发重规划（最多 3 次），3 次仍失败则降级到 ReActAgent 作为 safety net。该模块通过 `tests/agent/test_dag_validator.py` 的 5 类失败模式单测全部绿灯。

**新基线与测试集扩展**：实现 `src/agent/new_baselines.py`，其中包含 CoT（追加 "think step by step" 指示）、Plan-and-Solve（两阶段 LLM 调用，先自然语言 plan 再 ReAct 执行）、Reflexion（失败后生成反思追加到下一轮 system message）三个业界常用基线。测试集由 46 题扩展至 54 题，新增 8 道定向 hard 题，其中 4 道压 RATS 的工具区分力、4 道压 DAGPlanner 的多步依赖。

### 4.4 全量评测结果

对 9 配置 × 54 题共 486 次 run 的全量评测（`outputs/eval/20260423_103843/`，总耗时约 77 分钟）结果如下：

| 配置 | 成功率 | 平均迭代 | 平均工具调用 | 平均耗时(秒) |
|------|-------:|--------:|------------:|------------:|
| B0 (无工具) | 48.1% | 1.00 | 0.00 | 9.04 |
| B1 (minimal + 工具) | 90.7% | 3.00 | 2.65 | 5.35 |
| B2 (CoT) | 79.6% | 2.93 | 2.20 | 5.23 |
| B3 (Plan-and-Solve) | 92.6% | 4.07 | 3.76 | 8.07 |
| B4 (Reflexion) | 94.4% | 3.02 | 2.67 | 7.07 |
| **Ours (v4 prompt)** | **100.0%** | 3.46 | 2.46 | 15.12 |
| Ours + RATS | 90.7% | 3.44 | 2.46 | 9.69 |
| Ours + DAG | 94.4% | 5.94 | 4.96 | 11.17 |
| **Ours_full** | **100.0%** | 5.89 | 4.89 | 14.68 |

关键结论：主力方案 Ours 与 Ours_full 在本测试集上均达到 100% 成功率，相比朴素 ReAct 基线 B1 提升 10.3 个百分点，相比业界最强基线 Reflexion 提升 5.6 个百分点，达到预期目标。值得注意的是，B2 CoT 相比 B1 反而下降 11.1 个百分点，分析发现其冗长的中间推理文本更容易触发 LLM function.arguments 的 JSON 解析错误，此现象可作为论文中"只加 CoT 不做结构化引导存在局限"的佐证。RATS 单独接入时的效果是典型的效率换精度，在当前 12 工具规模下平均耗时降低约 36%，成功率相应损失 9.3 个百分点，该权衡在工具库扩展到百级规模时将反转为 RATS 的优势。

### 4.5 版本控制与可复现性

项目全过程使用 Git 管理，托管于 GitHub 私有仓库。每一轮评测的 `results.json` 与 `summary.md` 均按时间戳保存于 `outputs/eval/` 目录下，便于多轮实验间的 delta 分析。所有评测用例以 JSON 定义于 `tests/data/test_cases.json`，`scripts/run_eval.py` 提供命令行接口，支持按配置、按用例粒度的任意组合运行，保证全部实验结果可一键复现。

## 五、存在问题及预期进展

### 5.1 存在问题

从验证强度看，自建科学任务集虽已较开题阶段的设想明显扩充，但相对公开大规模 agent 基准仍属中小规模，外部可推广性需要更谨慎地表述。此前曾评估将 ScienceAgentBench、BFCL、GAIA 等公开数据纳入对比，因接口对齐与任务定义差异带来的工程成本较高而暂缓，当前以定向压力题补强弱项属于务实折中，但也会在论文中作为方法适用边界加以说明，并在条件允许时探索引入体量可控的公开子集以增强对照说服力。

从模型与工具生态看，主力实验依托同一供应商的 LLM 与 embedding，虽然便于统一鉴权与复现实验，但也意味着跨厂商、跨版本、跨模态的泛化结论仍不充分。RATS 与 DAGPlanner 在更强或更弱的基座模型上是否保持同样的相对优势，需要额外实验才能支撑，而这受到 API 成本与工程时间的现实约束。与此同时，科学工具库目前覆盖数值、统计与可视化三类高频能力，RATS 在"百级工具库"场景下的收益论证仍主要依赖外推，尚缺乏同口径的大规模工具库压力测试。

从形式化与评测口径看，DAGPlanner 的 Plan Schema 以 DAG 表达步骤依赖，天然适合链式、可拆分为明确子目标的科学计算流程，但不擅长表达需要显式循环或强分支探索的策略；这并不否定其在常见科学任务中的价值，却限制了方法叙事的普适性边界。评测侧使用数值、文件与工具覆盖等自动判分指标可以高效量化，但也可能把"合理解法却未按检查器要求调用某工具"的情况判为失败，例如某些步骤在代数上可化简为开方而绕开 `equation_solver`，从而暴露规则边界与智能体自主推理之间的张力。此类现象需要在讨论章节解释清楚，避免把检查器约束误写为"科学真理性"。

综合而言，以上问题并不否定当前工程与实验产出的可复现性，但要求论文在贡献陈述上更加克制，在局限性与未来工作中写得更具体，也更能回应开题阶段导师与专家关于"技术深度与评估充分性"的关切。

### 5.2 预期进展

后续工作将主要围绕"把证据链写完整"展开，而不是再做大范围功能扩张。写作上优先完成与两大算法模块直接相关的章节，使每个关键结论都能对应到可定位的实现文件、可复现的脚本入口与可核对的评测快照。图表上会把系统结构、方法流程、对比结果与关键案例组织成同一条阅读路径，减少只给结论不给出处的情况。定稿前将安排多轮通读，重点检查术语统一、指标口径一致、以及表格与文字描述不打架。

材料归档将同步推进，使源代码、测试数据、评测输出与说明性文字能够相互指向，满足学院对毕设材料完整性的要求。答辩准备以讲清问题定义、方法要点、实验设置与主要失败模式为目标，用少量高质量示意图支撑叙述，避免堆叠未解释的曲线与数字。

总体上，工程与主实验的阶段性目标已达到可支撑论文写作的程度，当前风险更多来自表达与组织：如何把复杂系统讲清楚、把贡献与局限讲准确。预期在后续阶段把论文与配套材料推进到可提交、可答辩的状态，并在终稿中明确哪些结论由本数据集直接支持、哪些属于合理外推与开放问题。
