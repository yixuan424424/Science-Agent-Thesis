\section{实验}

本章在 \autoref{eq:eval-criteria} 给出的三维判分定义之下，从主实验、按类别细分、消融研究、效率分析四个角度对本文方法进行实证评估。
所有数字均可在 \texttt{outputs/\allowbreak eval/\allowbreak 20260518\_120256/} 目录下的 \texttt{results.json} 与 \texttt{summary.md} 中追溯到原始落盘数据，
对应的复现命令为 \texttt{python scripts/\allowbreak run\_eval.py --configs b0 b1 b2 b3 b4 ours ours\_rats ours\_dag ours\_full}。

\subsection{实验设置}\label{sec:exp-setup}

\subsubsection{基座模型与超参数}

主实验使用通义千问 \texttt{qwen-turbo}（OpenAI 兼容接口），temperature 默认 0.2，单轮最大 token 数 4096，迭代上限 $M=10$。
RATS 的嵌入接口使用 \texttt{text-embedding-v3}，维度 1024，余弦相似度计算在内存中以 64 位浮点完成。
DAGPlanner 的硬上限设为最大重规划次数 $R=3$、单步最大参数修复次数 $F=2$、最大计划步数 8。
所有 LLM 调用均通过统一的 \texttt{LLMClient} 类，启用指数退避重试与 60 秒超时控制。

\subsubsection{测试集构成}

测试集包含 89 道科学任务，分布如\autoref{tab:cases-dist}所示。
相较于方法设计阶段使用的初版 54 题，本次评测在保持原有类别比例的前提下，额外补充了 35 道以 hard 和 vhard 难度为主的新题，
使整体难度分布向高难度倾斜（hard 占 46\%，vhard 占 6\%），以更严格地压测各方法在复杂场景下的表现。
全部用例以 JSON 形式定义于 \texttt{tests/data/test\_cases.json}，附录 C 给出扩展压力题的完整清单。

\subsubsection{评测指标}

主指标为成功率，即 \autoref{eq:eval-criteria} 三维判分同时通过的用例比例。
辅助指标包括平均迭代次数（每次运行中 LLM 与工具交互的轮数）、平均工具调用次数（每次运行中 \texttt{tool\_calls} 的次数，含失败重试）、平均端到端耗时（含 LLM 调用、工具执行与本地数据搬运）。
四个指标共同刻画方法在"能否解决"、"需要多少推理"、"需要多少工具"、"需要多少时间"四个维度上的代价收益。

主实验的几条关键观察如下：
\texttt{O1--O4 提示工程规则贡献显著}。
\texttt{Ours} 与 B1 唯一差异是 4 条提示规则的启用，二者成功率分别为 91.0\% 与 83.1\%，规则集贡献 7.9 个百分点，且不增加额外工具调用次数（Ours 平均 2.94 次 vs B1 平均 2.85 次，几乎持平）。
\texttt{B2 CoT 略有下滑但差距收窄}。
B2 比 B1 下降 1.1 个百分点（82.0\% vs 83.1\%），CoT 导致 JSON 解析失败的现象仍然存在，但在本实验中影响幅度有限（参见 \autoref{sec:analysis-cot-paradox} 的深入剖析）。
\texttt{B3 Plan-and-Solve 表现不如预期}。
B3 以 80.9\% 居所有有工具配置末位，原因在于 \texttt{qwen-turbo} 在两阶段分离的计划-执行框架下更容易在复杂任务上出现计划与执行脱节（参见按类别细分中 \texttt{composite} 类的具体数据）。
\texttt{DAGPlanner 与 RATS 单独接入均未能超越基线，体现了对基座能力的依赖}。
\texttt{Ours+RATS} 与 \texttt{Ours+DAG} 均与 B1 持平于 83.1\%，说明 RATS 的工具筛选截断与 DAGPlanner 的结构化规划在 \texttt{qwen-turbo} 下均未能有效发挥；
而两者组合的 \texttt{Ours\_full}（85.4\%）优于各自单用，表明两组件在一定程度上形成了互补，
DAGPlanner 在执行期捕获 RATS 误剪导致的工具不可用错误并触发 ReAct 安全网，使系统相比单独使用 RATS 或 DAGPlanner 有所恢复；
但整体仍低于不带算法增强的 \texttt{Ours}，进一步说明当前 RATS 和 DAGPlanner 设计的最大受益者是具有更强规划能力的基座模型。

类别细分的结论可以归纳为以下四点：
第一，\texttt{composite} 类别最能区分方法优劣。
该类别共 36 题（占总题量 40\%），以多步依赖链为主；
\texttt{Ours} 以 89\% 显著领先，而最强有工具基线 B4 仅 75\%；
这与本文"长依赖链科学任务正是朴素 ReAct 与自然语言 Plan-and-Solve 薄弱点"的论断一致。
第二，\texttt{numerical} 在所有有工具配置下均达到 100\%，说明单步数值计算已被现有工具体系充分覆盖，此类任务对方法设计的区分度有限。
第三，\texttt{statistics} 类别出现了 \texttt{Ours} 略低于 B1 的异常（81\% vs 88\%）。
分析失败用例发现，部分统计类题目要求严格遵守符号约定（如 t 统计量的正负号）或精确手动计算合并标准差后的派生量（如 Cohen's d），
在 \texttt{qwen-turbo} 这类较弱的基座上，v4 prompt 的规则约束有时反而导致模型过度思考后输出格式不符合判分期望；
这也说明提示工程规则在不同任务类型和基座模型上存在非均匀的效果，是未来需要针对具体类型做规则精细化的方向。
第四，\texttt{error\_recovery} 与 \texttt{visualization} 验证了稳健性。
\texttt{Ours} 在 \texttt{error\_recovery} 上达到 100\%，在 \texttt{visualization} 上与基线持平于 92\%；
对照 B0 在 \texttt{visualization} 上为 0\%（完全无法生成图表文件）、在 \texttt{error\_recovery} 上 90\%（部分题目能靠推理通过但缺乏工具），
工具库的覆盖范围是可视化类任务的必要条件。

消融实验揭示了三条关键结论：

\begin{itemize}
    \item \textbf{O1--O4 prompt 贡献 $+7.9$ pp，且效率代价最小}：B1 $\to$ Ours 即可观察到这一提升，而平均工具调用次数几乎不变（2.85 vs 2.94），端到端耗时仅增加 0.84 秒。提示工程因此是本文方法中性价比最高的组件。
    \item \textbf{RATS 与 DAGPlanner 单独接入均拉低成功率，揭示基座模型依赖性}：\texttt{Ours+RATS} 相比 \texttt{Ours} 下降 7.9 pp，\texttt{Ours+DAG} 同样下降 7.9 pp。前者源于 \texttt{qwen-turbo} 在工具精选后更难从有限候选集中正确选择；后者源于 DAGPlanner 的结构化规划需要更强的 JSON 生成和依赖推断能力，\texttt{qwen-turbo} 的规划失败率偏高，即使有 ReAct 安全网，补救率也有限。
    \item \textbf{RATS + DAGPlanner 组合有限恢复，体现互补但存在瓶颈}：\texttt{Ours\_full}（85.4\%）优于各自单用（83.1\%），说明两组件在错误捕获层面确实形成互补——DAGPlanner 在执行期吸收 RATS 误剪产生的工具不可用异常并触发 fallback；但整体仍低于不带算法增强的 \texttt{Ours}，进一步验证了这两个组件的价值发挥有赖于基座模型的规划能力达到一定水准。
\end{itemize}


\subsection{效率分析}\label{sec:exp-efficiency}

\autoref{fig:scatter} 把 9 配置的"平均工具调用次数"与"平均端到端耗时"绘成散点图，点的大小代表成功率。
散点图把方法选择的多目标权衡空间可视化：
B1（左下）是"低工具调用、低耗时但成功率欠缺"的典型；
Ours（左中）以几乎与 B1 持平的工具调用次数（2.94 vs 2.85）和适中的耗时（4.98 s），实现了最高的成功率（91\%），是全图 Pareto 最优点；
Ours\_full（右上）工具调用次数与耗时均最高，但成功率（85.4\%）反而低于 \texttt{Ours}，说明在 \texttt{qwen-turbo} 下为结构化规划付出的额外开销并未换来性能提升，属于当前基座能力瓶颈下的次优选择。

\begin{figure}[ht]
    \centering
    \includegraphics[width=0.85\linewidth]{fig_scatter.pdf}
    \caption{\label{fig:scatter}9 配置在"平均工具调用次数"与"平均端到端耗时"维度上的分布。点的大小正比于成功率。橙色为本文方法。}
\end{figure}

进一步审视 \autoref{tab:main-results} 中"仅在通过用例上"的耗时差异，可得三条额外结论：
第一，\texttt{Ours} 是全图成功率-耗时综合最优的配置（91\%，4.69 s），适合对精度和延迟同时有要求的生产场景；
第二，\texttt{Ours+DAG} 平均迭代次数最高（6.64 次），因为 DAG 显式 \texttt{Plan + Validate + Execute + Answer} 至少占用 4 轮，但产出的执行轨迹结构更规整、依赖关系可追溯；
第三，\texttt{Ours+RATS} 在通过用例上耗时为 5.73 s，虽然因误剪导致整体成功率与 B1 持平，但其端到端延迟仍低于 \texttt{Ours+DAG} 与 \texttt{Ours\_full}，在工具库扩张到百级规模时，RATS 的上下文压缩价值预计将超过当前的误剪代价。
